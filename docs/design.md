## Runtime Topology (authoritative, single story)

**One service answers both the API and the frontend on one TLS endpoint.** There is no separate frontend container and no reverse proxy.

- Long-lived services in `docker-compose.yml`:
  - `db` — PostgreSQL 16 with persistent volume `terraops-db`.
  - `app` — the Actix-web binary, built by `Dockerfile.app` (multi-stage: stage 1 `trunk build --release` produces `crates/frontend/dist/`; stage 2 `cargo build --release -p terraops-backend`; final stage copies both so the binary can serve `dist/` as static files). Exposes `:8443` (TLS) to the host.
- Profile service:
  - `tests` — built by `Dockerfile.tests` (Rust + Node + Playwright browsers pre-installed, no network use); orchestrated only by `./run_tests.sh`.
- TLS termination is done by the Actix-web binary via Rustls on port `8443`. The server cert (and optional client-CA) live on the Docker volume `terraops-runtime`, generated on first boot by `scripts/dev_bootstrap.sh`. No `.env` files.
- HTTP request routing inside `app`:
  - `/api/v1/**` → REST handlers.
  - `/assets/**` → hashed static artifacts from `dist/assets/`.
  - anything else → `index.html` (SPA fallback) so client-side `yew-router` paths work on refresh.
- Access contract: `https://localhost:8443/` (self-signed dev cert on first boot). Same origin serves both UI and API → no CORS surface.
- `./run_app.sh` wraps `docker compose up --build` and waits on `/api/v1/health`.
- `./init_db.sh` runs `docker compose run --rm app terraops-backend migrate` then `scripts/seed_demo.sh`.
- `./run_tests.sh` uses the `tests` profile to bring up `db` + a test-mode `app` and runs the full suite.

This single topology governs the scaffold, the Docker files, the file tree in `plan.md`, and the access contract.

## Product Overview

A fully offline single-node operations portal that fuses three domains — environmental performance tracking, product master-data governance, and talent matching — behind a role-aware Yew UI and an Actix-web REST backend running in one Docker service. It serves the host workstation and, when enabled, a restricted internal LAN via TLS + mTLS device pinning.

## System Overview

- **What**: single Rust binary (Actix-web) that serves both the Yew SPA and the REST API over TLS, backed by Postgres and local file storage, with in-process schedulers for reports/alerts/retention/notifications. No outbound network.
- **Core objective**: authoritative, offline, explainable intelligence across environment, catalog, and talent with audit-grade governance.
- **Shape**: Browser → TLS (+ optional mTLS device pin) → Actix-web `app` → Postgres + local object store. Background jobs run in-process. Mailbox `.mbox` export is the only "email" path.

## In-Scope Domains

1. Identity, RBAC, sessions, admin console.
2. KPI dashboards (cycle time, funnel stages, anomaly counts, efficiency trends) with slice-and-drill by site/department/time/category.
3. Configurable alert rules with duration-aware evaluation.
4. Scheduled + on-demand reports (PDF/CSV/XLSX) with durable job lifecycle.
5. Product master data (SPU/SKU, barcodes, categories, brands, units, state tax rates, shelf life, images, on/off-shelf).
6. Bulk CSV/XLSX import/export with preview, row validation, commit, immutable change history.
7. Environmental observations + analyst-configurable fusion (sources, alignment rules, confidence, windows, formulas).
8. Derived metrics (15-min MA, rate-of-change, comfort index) with "why this value" lineage.
9. Talent/role records, intelligent search, explainable recommendations, preference weights, private watchlists, feedback, cold-start rule.
10. Local notification center: subscriptions, read/unread, retry queue, `.mbox` file export.
11. Retention policy engine.
12. Admin monitoring: API latency, error rate, client crash reports.
13. Image signed-URL service, endpoint allowlist, and mTLS client-cert pinning admin surface.

## Explicit Out-Of-Scope

- Outbound SMTP or real email delivery. Only local `.mbox` artifacts.
- Cloud sync, SaaS analytics, any telemetry leaving the box.
- Multi-tenant architecture.
- Real-time push protocols (WebSocket/SSE) beyond polling the notification endpoint.
- Mobile-native clients.
- External secret managers / SSO / OIDC.

## Actors And Roles

**The five workspace names from the prompt are preserved verbatim:** Administrator, Data Steward, Analyst, Recruiter, Regular User. The app shell, nav, and role-gated pages use exactly those five names.

| Role | Scope |
|---|---|
| Administrator | Users, roles, retention, alert rules, allowlists, mTLS pins, monitoring, audit log |
| Data Steward | Product master data, imports, exports, change-history review |
| Analyst | Environmental sources, fusion definitions, lineage review, scheduled reports |
| Recruiter | Talent search, recommendations, preference weights, private watchlists, feedback |
| Regular User | Operator-level read of dashboards, alerts, notifications |

**Hiring managers** are a distinct real-world actor and are treated as first-class users of the recruiter-facing workspace. They are not collapsed into the Recruiter role label; instead, administrators grant them the `recruiter` role (optionally alongside other roles) because the Recruiter workspace is the product surface the prompt defines for talent work. Every RBAC and audit surface records the individual user, not a generic "recruiter" identity, so hiring-manager activity remains traceable.

Users may hold **multiple roles**; authorization is the union of each role's permissions.

## Actor Success Paths

- **Administrator** → login → admin console → create users & assign roles → tune retention → issue device certs + manage pin set → review monitoring → export audit log.
- **Data Steward** → login → Products workspace → upload CSV/XLSX → row-validated preview → commit → history reflects batch + actor → export filtered subset.
- **Analyst** → login → Environment workspace → create fusion definition (sources + alignment + confidence) → inspect derived metric chart → open "why this value" lineage panel → schedule weekly PDF report → observe durable job status.
- **Recruiter / Hiring Manager** (both use Recruiter workspace) → login → search candidates → review recommendations with match reasons → tune preference weights → add to private watchlist → thumbs-up/down → observe cold-start→blended transition after ≥10 feedback.
- **Regular User** → login → role-aware KPI dashboard → drill by site/department/time/category → acknowledge alert → read notifications.

## Architecture And Module Boundaries

Cargo workspace + operational scripts:

- `crates/shared` — DTOs, error codes, role enum, permission enum, pagination, timestamp helpers, validation traits. Consumed by backend and frontend.
- `crates/backend` — the single serving binary: Actix-web app, domain modules (`auth`, `rbac`, `users`, `ref_data`, `products`, `storage`, `metrics_env`, `kpi`, `alerts`, `reports`, `talent`, `notifications`, `monitoring`, `retention`, `security`), middleware, config, DB access, scheduler, TLS bootstrap, migrations, **static-serve of the frontend `dist/`**.
- `crates/frontend` — Yew SPA, router, context providers, API client (3 s timeout, single retry for idempotent GETs), role-aware pages per workspace. Built by Trunk; artifacts embedded into the `app` container at image-build time.
- `e2e/` — Playwright specs for synchronized cross-cutting flows.

Backend layering: `handlers/` → `service` → `repo` (sqlx) → `db`. Cross-cutting: `middleware/` (request-id, allowlist, 3 s budget, metrics, error-normalize), `errors.rs` (single `AppError` → JSON envelope), `config.rs` (one source of env reads), `telemetry.rs` (tracing).

## Authoritative Business Rules

1. **Pagination**: `page_size=50` default, max `200`. Offset-based. `X-Total-Count` header + `{items, page, page_size, total}` body.
2. **Request timeout**: backend enforces 3 s handler budget (→ 504). Frontend `fetch` uses 3 s timeout + exactly one retry for idempotent `GET`/`HEAD` only. Mutations never auto-retry.
3. **Comfort index (default)**: blend of temperature (°C), humidity (%), air speed (m/s); default formula and coefficients seeded and editable per `metric_definitions.formula_params`. Default alert rule: `CI > 82` sustained `> 15 min`.
4. **SKU on-shelf compliance (default)**: `on_shelf_count / active_count` rolling daily; default alert: `< 95%` for a full calendar day.
5. **15-min moving average**: trailing window; `≥ 3` samples required else `confidence_label=low`.
6. **Rate of change**: `(v_t − v_{t−Δ}) / Δ_minutes`, default Δ=5 min.
7. **Password policy**: min 12 chars, ≥1 per class; Argon2id params above. Lockout after 10 failures / 15-min rolling window.
8. **Tokens**: access JWT 15 min; refresh opaque 30-day idle / 90-day absolute; rotated per use (old revoked).
9. **Image URLs**: `sig = HMAC_SHA256(image_key, "{path}|{user_id}|{exp}")`; `exp ≤ 600 s`; single-use nonce optional.
10. **Uniqueness**: `users.username`, `users.email_hash`, `products.sku`, `products.barcode` (when present), `categories(name, parent_id)`.
11. **Import commit**: requires zero row-level errors; warnings allowed; commit is transactional.
12. **Change history**: immutable by DB trigger.
13. **Cold start ranking**: `feedback_count(user, role_scope) < 10` → `score = 0.6*recency + 0.4*completeness`. Otherwise blended: `0.35*match + 0.2*recency + 0.15*completeness + 0.3*preference_weighted_feedback`.
14. **Retention defaults** (configurable): env raw 18 months, KPI aggregates 5 years, inactive-user feedback 24 months, audit log indefinite.
15. **Scheduled reports**: cron validated; if previous run is still `running`, next trigger logs `skipped_prev_running`; one transient retry, then `failed`.
16. **Notification retry**: exponential 30 s → 2 m → 10 m → 1 h → 4 h; max 5 attempts, then `failed`.
17. **Timestamps**: persisted UTC TZ-aware; displayed MM/DD/YYYY hh:mm AM/PM in user-local TZ; formatting centralized in `shared::time`.
18. **Masking**: `email_mask` (e.g. `j***@e***.com`) precomputed at write. Plaintext email is **only** obtainable by Administrator and by the owning user, by decrypting `email_ciphertext` server-side.
19. **Allowlist**: if `endpoint_allowlist` has ≥1 enabled entry, requests whose source IP does not match any CIDR are rejected with 403 before authentication.
20. **Device pinning (mTLS)**: when `mtls_config.enforced = true`, Rustls requires a client certificate whose SPKI SHA-256 is present in `device_certs` (not revoked). Otherwise the TLS handshake is refused. When `enforced = false`, mTLS is not requested and the pin set is dormant (admin can still manage it).

## State Machines And Lifecycles

- **Import batch**: `uploaded → validated → committed | canceled`; `validated → failed` on commit error. No reopen of `committed`.
- **Report job**: `queued → running → success | failed`; `running → failed` retryable once; any non-terminal can `cancel`.
- **Alert event**: `firing → ack → resolved` (or `firing → resolved` when auto-clears).
- **Notification delivery attempt**: `pending → success | failed`; on failure, scheduler enqueues the next attempt up to 5.
- **Session**: `active → rotated | revoked | expired`. Rotation inserts a new row with `rotated_from`.
- **User account**: `active ↔ disabled`; `active → locked (auto) → active (admin unlock)`.
- **Device cert**: `issued → revoked`. Revocation is immediate; next TLS handshake refuses.

**Auth classification per endpoint** — every endpoint in `docs/api-spec.md` carries exactly one of these labels:

- `PUBLIC` — no auth (health, ready, login, refresh, SPA static).
- `AUTH` — any authenticated user; no permission required.
- `PERM(code)` — route-level permission check via `RequirePermission` extractor.
- `SELF` — authenticated user, object-level owner check in service layer (e.g., their own watchlist, weights, notifications, report job).
- `PERM_OR_SELF(code)` — pass if caller holds `code` OR is the owner (e.g., admin reading any user's audit entries, owner reading their own).

The service layer always performs the object-level check even when `PERM(...)` suffices at the route; permission is necessary but not sufficient when ownership is defined.

## Validation And Error Handling

- Input validation with `validator` on DTO structs; rejection → `422 VALIDATION_FAILED` with `{code, field_errors[]}`.
- Normalized envelope: `{error_code, message, request_id, details?}`.
- Error code → status: `AUTH_INVALID_CREDENTIALS` 401, `AUTH_LOCKED` 423, `AUTH_FORBIDDEN` 403, `VALIDATION_FAILED` 422, `NOT_FOUND` 404, `CONFLICT` 409, `RATE_LIMITED` 429, `TIMEOUT` 504, `MTLS_PIN_REJECTED` (TLS handshake, never reaches HTTP), `INTERNAL` 500.
- `>3 s` handler → `504 TIMEOUT` with `request_id`, row logged.
- No raw stack traces in responses; traces only in `tracing` JSON logs.

## Security, Compliance, And Data Governance

### Authentication & sessions

- Argon2id hashing with constant-time compare.
- Access JWT HS256 signed with `jwt_signing_key` (256-bit random, generated by `dev_bootstrap.sh` into volume `terraops-runtime/keys/jwt.key`; never committed).
- Refresh tokens opaque 256-bit random, stored as SHA-256 hash, rotated per use; refresh delivered via HttpOnly SameSite=strict cookie `tops_refresh`.
- CSRF: state-changing requests require `X-Requested-With: terraops` in addition to the bearer access token.

### Email & PII storage (authoritative)

- The only forms of a user's email stored in Postgres are:
  1. `email_ciphertext` — AES-256-GCM ciphertext (nonce prepended) encrypted with `email_enc_key` (256-bit, generated by `dev_bootstrap.sh` into `terraops-runtime/keys/email_enc.key`).
  2. `email_hash` — HMAC-SHA256 using `email_hmac_key` (separate 256-bit key in the same volume) for exact-match uniqueness and lookup. Unique index enforced.
  3. `email_mask` — precomputed display string (e.g. `j***@e***.com`).
- There is **no plaintext `email` column**. Inserts/updates go through `users::store_email(plain)` which computes all three.
- API exposure:
  - `GET /auth/me` and `GET /admin/users/{id}` return the caller's own full email (decrypted server-side) and any requested user's mask + full email **only to Administrator**.
  - All list endpoints expose `email_mask` only.
- Key bootstrap keeps the no-`.env` rule: `dev_bootstrap.sh` writes the three keys (`jwt.key`, `email_enc.key`, `email_hmac.key`, `image_hmac.key`) on first boot into the runtime volume. Production deployments replace the volume contents with operator-supplied keys; the repo never carries any of them.
- Tests required:
  - Unit: `aes_gcm` round-trip; HMAC stability; mask formatting.
  - HTTP: non-admin list endpoints return mask only; admin `GET /admin/users/{id}` returns full email; `GET /auth/me` returns full email for self.
  - DB-contents test: after creating users, a direct `SELECT` confirms `email_ciphertext` is non-plaintext bytes and no column in `users` contains the plaintext substring.

### TLS + mTLS device pinning (replaces prior SPKI-in-browser idea)

- Server TLS always on (`:8443`). Server cert auto-generated on first boot; operators may mount their own.
- **Device pinning is server-enforced mTLS**, not browser SPKI verification:
  - Admin operates a `device_certs` registry. The `scripts/issue_device_cert.sh` helper (run by the admin inside the container) generates a client-cert signed by the runtime's internal CA (also auto-generated) and emits a `.p12` bundle to the admin on the runtime volume. The client's SPKI SHA-256 is recorded in `device_certs`.
  - The operator installs that `.p12` on each approved device/browser profile.
  - When `mtls_config.enforced = true`, Rustls is configured with `with_client_cert_verifier(PinnedSpkiVerifier)` which:
    1. Requires the TLS handshake to include a client certificate.
    2. Computes the SPKI SHA-256 of the presented cert.
    3. Looks up that hash in the in-memory pin set (reloaded on every admin mutation of `device_certs`).
    4. Fails the handshake (not the HTTP request) if absent or if `revoked_at` is set.
  - When `mtls_config.enforced = false`, client certs are not requested and the portal behaves as one-way TLS; admins may prepare the pin set before flipping the enforcement switch.
- Admin API (matches the inventory in `docs/api-spec.md §Security & Ops`):
  - `GET /api/v1/admin/device-certs` — list the pin set.
  - `POST /api/v1/admin/device-certs` — register a pre-issued client-cert SPKI (label + user assignment).
  - `POST /api/v1/admin/device-certs/{id}/revoke` — revoke a pin (dedicated endpoint so revocation is a first-class state transition, not a deletion side-effect).
  - `DELETE /api/v1/admin/device-certs/{id}` — remove a pin-set record entirely.
  - `GET /api/v1/admin/mtls-config` — read enforcement state.
  - `PUT /api/v1/admin/mtls-config` — flip `enforced`.
  - Pin set mutations bump an in-memory generation counter; the verifier reloads within one second.
- Admin UI: `pages/admin/device_certs.rs` + `pages/admin/mtls.rs` show the pin set, enforcement state, and instructions for installing the issued bundle.
- Tests:
  - Integration spins up the server with an ephemeral CA, issues two client certs, pins one, sets `enforced=true`, and proves: (a) pinned client connects; (b) unpinned client's TLS handshake is refused; (c) revocation takes effect within one second.

### Other controls

- Sensitive fields (password hash, email ciphertext, HMAC keys, refresh token hash) are redacted from logs via `tracing` field filters.
- Retention enforced hourly against `retention_policies`.
- Audit log append-only (trigger).
- Signed image URL verifier checks user_id + expiry + signature.
- `endpoint_allowlist` is the IP-layer gate; mTLS is the device-layer gate. Either may be enabled independently.

## Offline, Queueing, Reliability, And Background Jobs

- `tokio_cron_scheduler` hosts: **report scheduler**, **alert evaluator** (every 30 s, duration-aware), **notification retry worker** (every 15 s), **retention enforcer** (hourly), **metrics aggregator** (hourly rollup after 7 days).
- All jobs idempotent (state + `attempts` guards). All in-progress state persisted; restart resumes from DB.
- Frontend tiered cache: in-memory LRU + IndexedDB for reference lookups (categories, brands, units, sites, departments, perm bitset) and signed-image thumbnails. Incremental loading for long lists via scroll sentinel + offset paging. Explicit placeholder states: `loading`, `empty`, `stale-cache (as-of …)`, `error (retry)` — never silently show stale data as fresh.

## Reporting, Analytics, Search, And Exports

- KPIs served primarily from `kpi_snapshots` (refreshed every 5 min) + on-demand drill against operational tables.
- Report runner: `askama` HTML template → `printpdf` PDF; CSV via `csv`; XLSX via `rust_xlsxwriter`. Artifact path + SHA-256 recorded.
- Export endpoints stream with correct MIME and filename.
- Talent search: candidate `tsv` over `full_name || skills || major`; query supports keywords, skill tokens, major, education level, experience band, availability. Scoring in Rust combines `ts_rank_cd`, trigram similarity, and structured filters.
- Recommendations: cold-start vs blended per §Authoritative Business Rules. Reasons returned as structured list (`matched_skills`, `experience_fit`, `availability_match`, `recency_days`, `completeness_pct`).

## Verification Strategy

**Authoritative threshold sentence (single source of truth, referenced verbatim by `docs/test-coverage.md` and `repo/plan.md`):**

> The 90% line-coverage floor applies **only to `crates/shared` + `crates/backend`** (Rust native code). The `crates/frontend` Yew/WASM crate has its own **separately enforced 80%** line-coverage floor. Playwright contributes flow verification only and **does not contribute to either line-coverage number**.

Canonical commands (used identically in `docs/test-coverage.md` and invoked by `./run_tests.sh`):

```bash
# Gate 1 — backend + shared ≥ 90% (single command generates LCOV and enforces floor)
# Scope is restricted to the two native Rust crates via explicit -p flags; the frontend
# crate is never compiled under this invocation, so no WASM or frontend source can
# contribute to the rust.lcov numerator or denominator.
cargo llvm-cov --no-fail-fast \
  -p terraops-shared -p terraops-backend \
  --ignore-filename-regex '/(tests|migrations|\.sqlx)/' \
  --fail-under-lines 90 \
  --lcov --output-path coverage/rust.lcov

# Gate 2 — frontend WASM ≥ 80%
RUSTFLAGS='-C instrument-coverage' \
LLVM_PROFILE_FILE='target/wasm32-coverage/cov-%p-%m.profraw' \
cargo test -p terraops-frontend --target wasm32-unknown-unknown
grcov ./target/wasm32-coverage \
  --binary-path ./target/wasm32-unknown-unknown/debug \
  -s crates/frontend/src -t lcov --ignore 'main.rs' \
  --threshold 80 -o coverage/frontend.lcov

# Gate 3 — endpoint parity (see §Endpoint Audit Rules in docs/test-coverage.md)
scripts/audit_endpoints.sh

# Flow gate — Playwright (flow evidence only, not line coverage)
npx --prefix e2e playwright test
```

Gates 1, 2, and 3 all contribute to `./run_tests.sh`'s exit status. Gate 3's strictness is mode-driven (see §Endpoint Audit Rules) so scaffold-time runs remain deterministic while final delivery enforces 100% forward parity.

- **True no-mock HTTP** tests are required for every endpoint in `docs/api-spec.md` and are what Gate 3 verifies — via ID-based matching between the endpoint inventory and `t_<id>_*` test function names in `crates/backend/tests/http/**`.

## Assumptions, Dispositions, And Open Items

- **Hiring managers**: first-class actor; they use the Recruiter workspace because the prompt defines exactly five workspace names. Admins assign them the `recruiter` role (and other roles as needed). Individual identity preserved in audit log; no generic-role collapse.
- **Timezone default**: `America/New_York` (12-hour display is US-typical); per-user override.
- **Email at rest**: authoritative model is AES-256-GCM ciphertext + HMAC hash + precomputed mask, keys in `terraops-runtime` volume. Candidate email is stored as mask only (not encrypted), because TerraOps does not collect plaintext candidate email at this phase.
- **Device pinning**: authoritative model is mTLS with SPKI-hash pin set enforced by Rustls, managed by admin. Browser-side pinning is explicitly out of scope.
- **Open**: none blocking.
